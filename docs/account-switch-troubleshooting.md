# アカウント切りかえ時の IndexedDB と OAuth の不調まとめ

> ※プログラミング初心者の同級生に説明するノリで書いてるよ。難しく感じたら、用語をググりながらゆっくり読んでみて！

## 何が起きていたの？

1. **IndexedDB が昔のアカウント用 DB をそのまま使い続ける問題**
   - Classroom の URL には `authuser=0` みたいなパラメータが付いてて、今まではその数字だけで DB 名を作ってたんだ。
   - でも Google のアカウントを入れ替えるときって、同じ `authuser` 番号のまま別の人に入れ替わることが普通にある。
   - だから番号だけを頼りにすると、「別人なのに同じ IndexedDB ファイルを共有しちゃう」という事故が発生。結果として古い課題データが残ったままになるよ。

2. **OAuth の再認証が出てこない問題**
   - Chrome 拡張の `chrome.identity.getAuthToken` は、一度トークンを取るとキャッシュに保存してくれる親切仕様。
   - ただしアカウントを切り替えても、拡張が「別人になった！」って気づかなければ、前のトークンをそのまま出し続けちゃう。
   - そのせいで OAuth の同意画面が表示されず、Google Classroom API も古い人の情報のまま動いちゃう。

## 今回の対策

1. **DB 名にアカウントの指紋をくっつけた**
   - GAIA ID やメールアドレスを直接保存すると危ないから、`hashString` でハッシュ化した「指紋」を作ったよ。
   - これを `u0-gxxxx` みたいな形で IndexedDB 名にくっつけることで、別アカウントなら必ず別 DB になるようにした。

2. **指紋が変わった瞬間に OAuth トークンを全消去**
   - コンテンツスクリプトから「この指紋のアカウントを使ってるよ！」ってヒントを飛ばす。
   - バックグラウンド側では、指紋が前回と違ったら `clearAllCachedAuthTokens` を走らせてキャッシュを空っぽにする。
   - これで次の API 呼び出し時にちゃんと OAuth 同意を求めてもらえるし、トークンの取り違えもなくなる。

## これからの運用ポイント

- 新しいアカウントでサインインした直後にデータが古かったら、まず指紋キーが正しく取れているか（GAIA ID またはメールが DOM から拾えているか）を確認してね。
- もしドメイン変更などで GAIA ID が隠された場合は、メールアドレスのパターン抽出か、別のメタタグを探してハッシュ材料を更新しよう。
- OAuth が再表示されないケースを見つけたら、バックグラウンドのログ（`[GCX] Cleared cached OAuth tokens`）が出ているかチェック。出ていなければ指紋の送信を疑って。

これでアカウント切り替え時の「前の人のデータが残る」「OAuth が出ない」っていうモヤっとをまとめて解決できるはず。困ったらこのメモをまた見に来てね！
